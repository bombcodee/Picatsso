2. AI 프로세스 — 파일별 역할

2번의 AI 호출이 어떤 파일을 거쳐서 진행되는지 보여줍니다.

1
성격 분석 (텍스트 → JSON)
모델: gemini-2.5-flash
use-cat-analysis.ts 택배기사 — 사용자 입력을 FormData로 묶어서 서버에 전달
api/analyze/route.ts 접수처 — FormData 받아서 이미지를 base64로 변환, GeminiAnalyzer에 넘김
gemini-analyzer.ts 실행자 — ANALYSIS_SYSTEM_PROMPT + 사진 + 입력값을 Gemini에 보냄
prompts.ts 레시피 — ANALYSIS_SYSTEM_PROMPT 제공 ("고양이 행동 전문가로서 분석해줘")

결과: { temperament: "curious", loves: ["츄르"], keywords: [...], ... } ← 이 JSON이 2차 호출의 재료가 됨

2
이미지 생성 (텍스트 → 그림)
모델: gemini-2.5-flash-image
use-image-generation.ts 택배기사 — 1차 분석 결과 + 장면 설명을 서버에 전달
api/generate/route.ts 접수처 — 분석결과에서 ArtStyle 매칭, GeminiImageGenerator에 넘김
prompt-builder.ts 레시피 조합사 — 6가지 재료를 빈칸에 끼워 넣어 최종 프롬프트 완성 (아래 상세)
gemini-image-generator.ts 실행자 — 완성된 프롬프트를 Gemini Image에 보내고 그림 2장 받음

3. 프롬프트 조립 — 레고 블록 구조

prompt-builder.ts가 여러 독립 블록을 하나의 최종 프롬프트로 조립합니다. 각 블록은 독립적으로 교체 가능.

IMAGE_GENERATION_TEMPLATE (조립 틀)

플레이스홀더({cubismBase}, {moodKeywords} 등)가 있는 영어 템플릿. prompt-builder가 빈칸을 채움.

공통 베이스 CUBISM_BASE_BLOCK — 피카소 큐비즘 화법 (굵은 윤곽선, 면 분할, 다중 시점). 모든 그림에 항상 적용.
과학 연결 CAT_PICASSO_CONNECTION — 고양이 시각의 과학적 사실 (시야각 200°, 이색형 색각, 야간시력)이 왜 큐비즘과 맞는지 설명.
성격 악센트 moodKeywords + characteristics — 성격별 분위기 키워드. personality-mapping.ts에서 가져옴. (아래 테이블 참고)
사용자 입력 sceneDescription, personalityType, keywords — 장면 설명, AI가 분석한 성격 유형명, 성격 키워드
감정 규칙 loves / dislikes — 파레이돌리아 효과 + 실제 존재 시 강조. 4가지 규칙이 템플릿에 고정.

조립 순서 (prompt-builder.ts)

1단계: 공통 블록 삽입 {cubismBase} → CUBISM_BASE_BLOCK, {catPicassoConnection} → CAT_PICASSO_CONNECTION
2단계: 성격 + 장면 삽입 {artistLabel}, {characteristics}, {moodKeywords}, {sceneDescription}, {temperament} 등
3단계: 감정 표현 삽입 {loves}, {dislikes} → 분석 결과의 emotionalColorMap에서 가져옴

4. 성격별 악센트 — 7가지 기질

큐비즘 베이스는 동일. 성격에 따라 분위기·에너지·색감만 달라짐. (C안 하이브리드 전략)

기질 사용자 라벨 분위기 핵심 영어 키워드
playful 장난꾸러기 화가 밝고 역동적, 기하학적 장난기 vibrant energy, geometric playfulness
fierce 열정의 화가 날카로운 각도, 강한 대비, 극적 구도 dramatic contrast, aggressive angles
calm 온화한 화가 부드러운 곡선, 따뜻한 톤, 여유 gentle curves, warm serenity
curious 몽상가 화가 왜곡된 형태, 몽환적, 다중 초점 dreamlike distortion, whimsical wonder
aloof 고고한 화가 거대한 볼륨, 조각적, 차가운 톤 sculptural grandeur, cool elegance
chaotic 자유로운 화가 파편화된 면, 콜라주, 다양한 질감 fragmented collage, textural chaos
melancholy 감성파 화가 블루톤 분위기, 길쭉한 형태, 고요 contemplative blue undertones, serene solitude

5. 레이어 구조 — 누가 누구를 부르나

위에서 아래로만 호출. 아래 레이어는 위를 모릅니다.

▼ fetch('/api/...')

analyze/route.ts 성격 분석 generate/route.ts 이미지 생성
▼ 함수 호출
services/ — 외부 API 어댑터
ai/gemini-analyzer Gemini 분석 image/gemini-image-generator Gemini 이미지 image/prompt-builder 프롬프트 조립 storage/file-handler 파일 처리
▼ 상수/타입 참조

6. 폴더 의미 사전

각 폴더가 존재하는 이유와 담당하는 역할.

src/ 하위

app/Next.js App Router. URL 경로와 1:1 매핑. page.tsx=화면, api/=서버 엔드포인트. 여기서 Gemini API 키 같은 민감 정보를 안전하게 사용.
components/화면에 보이는 모든 것. ui/=라이브러리 부품, common/=재사용 공통, layout/=페이지 껍데기, features/=화면 단위 그룹.
hooks/React 커스텀 훅. 화면(components)과 서버(api) 사이의 중간 다리. "버튼 클릭 → API 호출 → 결과 저장" 같은 프론트 로직을 담당.
services/외부 API와 통신하는 어댑터 레이어. 인터페이스(계약서)와 구현체(Gemini용)가 분리. 나중에 API를 바꿔도 여기만 수정.
lib/순수한 데이터 정의. API 호출이나 UI 코드 없음. 프로젝트 전역에서 쓰는 타입, 상수, 설정의 중앙 보관소.
lib/constants/변하지 않는 값들. 프롬프트 텍스트, 색상 팔레트, 성격→화풍 매핑. "프로젝트의 설정 파일" 역할. 여기만 수정하면 AI 결과물이 바뀜.
lib/types/TypeScript 타입 정의. CatAnalysis, ArtStyle, Temperament 등. 데이터가 어떤 모양인지 정의하는 "설계도".

docs/ 하위

docs/manual/기술 매뉴얼. 아키텍처, 데이터 흐름, API 스펙, 설정 가이드, 시각화 가이드(이 파일).
docs/prompt-management/프롬프트 전용 관리. 토의 기록(discussion-log), 화가 레퍼런스(artist-reference), 설계 문서(PROMPT_ENGINEERING).
docs/references/외부 참고자료. API 가격/스펙, 고양이 색각 과학 논문 정리.

8. 문서 지도 — 어떤 문서가 뭘 담당하나

프로젝트의 모든 문서와 각각의 역할.

프로젝트 관리

CLAUDE.md
Claude Code 지침. 등급 체계(규칙/가이드), 협업 흐름, 필수 참조 문서 목록. 이 파일이 바뀌면 visual-guide.html도 업데이트.
PRD.md
제품 요구사항 정의서. "뭘 만드는가 + 왜 만드는가". 확정된 기능만 기록.
ROADMAP.md
개발 로드맵 + 아이디어. "언제 만드는가". Phase별 태스크 + 💡 아이디어 섹션.
BACKLOG.md
작업 이력. 날짜, 작업 ID, 내용, 상태. 모든 완료/진행 작업의 기록.
MEMORY.md
세션 연속성. 마일스톤별 자동 업데이트. "이어서 하자" 할 때 여기부터 읽음.
CONVENTIONS.md
코딩 규칙. 모듈화 원칙, HTML 분리 기준, 어댑터 패턴 규칙.

기술 매뉴얼 (docs/manual/)

ARCHITECTURE.md
아키텍처. 전체 파일 트리, 폴더 의미, 레이어 구조, 어댑터 교체 가이드.
visual-guide.html
시각화 매뉴얼 (이 파일). 프로세스 흐름도, 프롬프트 조립, 모듈 관계를 그림으로.
DATA_FLOW.md
데이터 흐름도. 입력 → 분석 → 생성 전체 데이터 파이프라인.
API_SPEC.md
API 엔드포인트 상세 스펙. 요청/응답 형식, 에러 코드.
SETUP_GUIDE.md
환경 설정. 설치 방법, .env 설정, Vercel 배포 가이드.

프롬프트 관리 (docs/prompt-management/)

PROMPT_ENGINEERING.md
프롬프트 설계. 파이프라인, 매핑 테이블, 감정 규칙, 튜닝 로그.
discussion-log.html
토의 기록. 프롬프트 관련 토의 내용과 결정 사항. 태그별 필터.
artist-reference.html
화가 레퍼런스. 피카소 시기별 기법, 색상, 키워드. 탭 형식.
