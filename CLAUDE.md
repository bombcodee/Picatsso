# Picatsso - Claude Code 프로젝트 지침

## 필수 참조 문서
아래 docs/ 폴더의 모든 문서를 반드시 확인하고 작업에 반영할 것:

- `docs/PRD.md` — 제품 요구사항 정의서 (기능, 타겟, 플로우)
- `docs/ROADMAP.md` — 개발 로드맵 및 진행 상황 (PDCA 기반)
- `docs/CONVENTIONS.md` — 코딩 규칙 및 모듈화 원칙
- `docs/BACKLOG.md` — 개발 백로그 (작업 이력)
- `docs/TEST_SCENARIOS.md` — 테스트 시나리오
- `docs/BUG_REPORTS.md` — 버그 리포트
- `docs/references/REFERENCES.md` — 기술 참고자료 (API 스펙, 가격, 리서치 결과)
- `docs/references/cat_color_vision.md` — 고양이 색각 과학 레퍼런스
- `docs/prompt-management/PROMPT_ENGINEERING.md` — 프롬프트 설계 문서
- `docs/prompt-management/artist-reference.html` — 화가별 시기/기법/키워드 레퍼런스
- `docs/prompt-management/discussion-log.html` — 프롬프트 토의 로그 + 결정 사항
- `docs/MEMORY.md` — 세션 연속성 메모리 (진행 상황 요약)
- `docs/manual/ARCHITECTURE.md` — 아키텍처 + 파일 구조 + 어댑터 교체 가이드
- `docs/manual/DATA_FLOW.md` — 데이터 흐름도
- `docs/manual/API_SPEC.md` — API 엔드포인트 스펙
- `docs/manual/SETUP_GUIDE.md` — 환경 설정 가이드

## 문서 관리 흐름

```
ROADMAP 아이디어        ← 브레인스토밍, 자유롭게 아이디어 기록
       │
       ▼  (토의 후 채택 확정)
       │
PRD 기능 스펙에 반영     ← "이건 만든다" 확정, 스펙 작성
       │
       ▼  (확정된 스펙 기반)
       │
ROADMAP Phase에 태스크 배치  ← "언제 만든다" 실행 계획
```

| 문서 | 역할 | 성격 |
|------|------|------|
| `PRD.md` | **What & Why** — 제품 정의서 | 확정된 스펙, 채택된 기능만 기록 |
| `ROADMAP.md` | **When & How** — 실행 로드맵 + 아이디어 | 살아있는 문서, 자주 업데이트 |

- 🔒 사용자 승인 없이 PRD 수정 금지
- 🔄 토의 결과는 성격에 따라 적절한 문서에 정리 (ROADMAP, PRD, CONVENTIONS, REFERENCES 등)

## 협업 흐름

### 개발 진행 흐름
```
토의/플랜 → 🔒 사용자 승인 → 구현 → 🔒 빌드 확인 → 사용자 확인 → 수정 → 커밋/푸시
                 ↑                                       │
                 └────── 피드백이 있으면 돌아감 ───────────┘
```
- 🔒 사용자 승인 없이 개발 착수 금지
- 🔒 빌드 통과 확인 후 커밋 (깨진 코드 커밋 방지)
- 🔄 사소한 변경은 사용자 UI 확인 없이 커밋 가능

### 토의 흐름
```
질문/아이디어 → 심층 분석 + 옵션 제시 → 토의 → 승인 시 실행
                                            → 보류 시 적절한 문서에 기록
```
- 🔒 심층 분석 요청 시 구현하지 말고 토의 먼저
- 🔄 명확한 요청("이거 해줘")은 토의 없이 바로 실행 가능

### 테스트 흐름
```
🔒 빌드 통과 → 로컬 실행 → 사용자 UI 확인 → 피드백 → 수정 → 재확인
                                → 🔒 API 연동 테스트 → 프롬프트 튜닝 → 배포
```
- 🔒 API 연동 테스트 없이 배포 금지
- 🔄 UI 피드백은 반복 횟수 제한 없이 유연하게

### 커밋 흐름
```
커밋 요청 → 🔒 문서 정합성 검증 → 위험 요소 있으면 경고 + 승인 → 커밋 + 푸시
```
- 🔒 관련 문서(ARCHITECTURE, API_SPEC, MEMORY 등)가 코드와 일치하는지 확인
- 🔒 위험 요소(깨진 참조, 잘못된 스펙 등) 발견 시 반드시 알림
- 🔄 변경 없는 문서는 검증 생략 가능

## 작업 규칙

### 🔒 규칙 (절대 생략 불가)
1. 코드 작성 전 `docs/CONVENTIONS.md`의 모듈화 원칙 준수
2. 모든 외부 API는 어댑터 패턴으로 감싸서 교체 가능하게 구현
3. 사용자 허가 없이 개발 착수 금지
4. 모든 작업 완료 후, 요청 → 처리 결과 형식으로 요약
5. 사용자가 여러 질문/요청을 보냈을 때 빠짐없이 응답. 마지막에 자가 검증
6. 커밋 전 문서 정합성 검증 + 위험 요소 경고

### 🔄 가이드 (기본 수행, 상황에 따라 조정)
7. 작업 완료 시 `docs/BACKLOG.md`에 이력 기록
8. 버그 발견 시 `docs/BUG_REPORTS.md`에 기록
9. `docs/MEMORY.md`는 마일스톤 완료 시 자동 업데이트 + "메모리에 저장해줘" 시에도 업데이트
10. "이어서 하자" 요청 시 `docs/MEMORY.md` 읽고 → 현황 + 남은 플랜 제시 → 확인 후 진행
